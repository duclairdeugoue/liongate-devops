PALIGO Architecture - Where It Stands, Where We Can Take It

A working opinion, not a spec. Written to be argued with.
Based on: current architecture, domain-by-domain review, and a few hours of discussion with the team.

We went through the full domain-by-domain review. Rather than restate every finding, this is our take on the three things we actually care about - clarity, extensibility, scalability - and where the current architecture falls short of each.

1. What's improvable - five things, ranked

Pricing depends live on Odoo
Highest riskScalability + Extensibility

Storefront calls Pricing directly, bypassing the Middleware. Pricing in turn reads from Odoo - an accounting system, not a storefront-scale read API. Every filtered search result needs a price, so at 10x traffic this is 10x load hitting the ERP. It also quietly blocks Phase 3's dynamic pricing goal: you can't do sub-second margin adjustment on an ERP-driven read path.

Fix: Pricing owns its own fast read store, fed asynchronously off listing.price.changed events. Odoo becomes downstream (books the transaction), not upstream (source of live price).
Identity has two independent trust relationships
Medium risk, compoundingClarity + Extensibility

Storefront and Directus both talk to Keycloak directly. Not urgent today, but nothing stops the next consumer - merchant app, Logistix, whatever's next - from copying the same pattern, since there's no stated rule against it.

Fix: Centralize token validation at the Gateway now, while only two services need migrating, not five.
Storefront has three separate backend relationships
Medium riskClarity

Middleware, Keycloak, and Pricing - instead of one. This is the clarity problem in its purest form. A new engineer has to learn two different integration contracts for what should be a single consistent one.

Fix: Same fix as above - one Gateway/Backend For FrontEnd (BFF) as the only thing the frontend talks to. Solves this and the Pricing coupling at once.
Catalog and Merchant Listing data aren't separated
Medium riskExtensibility + Scalability

Generic taxonomy/barcode data and merchant-specific price/quantity both risk living in Directus. Taxonomy is read-heavy and rarely changes; listings change constantly. Mixed together, both get harder to scale and cache. It also blocks a confirmed real requirement - one product belonging to multiple taxonomy groups (sand for construction and winter roads) - which a single-parent model can't express without rework.

Fix: Split into two data owners now, before Phase 2's multi-group requirement forces a migration under deadline.
Analytics risk, if built the easy way
Lower urgency, cheap to prevent nowScalability

Not built yet, so this is a "don't" rather than a "fix." If Market Intelligence ends up querying Order/Pricing/Merchant databases directly for reports, it competes with production traffic - worst exactly when it matters most, during peak season.

Fix: Event-driven ingestion from day one. Never in the synchronous path of checkout or pricing.

3. Where We Can Take It

One Gateway as the single front door. Domain services own their data and publish events instead of getting queried live by whoever needs something. Payment and Logistics built as clean standalone services from day one, since both are already flagged for eventual commercialization - retrofitting a bounded service after the fact always costs more than starting with one. And country as a field on every domain now, even single-country, so Austria in Q4 is a config rollout instead of a migration.

Sequencing - What We Can Do in What Order

Order Move Why now, not later
1 Gateway consolidation (single entry point for Storefront) Cheapest fix, unblocks Identity fix and the Pricing decoupling at once
2 Pricing decoupled from live Odoo reads Highest current risk; also a hard prerequisite for Phase 3 dynamic pricing
3 Catalog / Merchant Listing data split Needed before Phase 2's multi-group taxonomy requirement lands
4 Payment & Logistics built standalone from the start Both are already slated for commercialization (Logistix SaaS) - build once, not twice

Bottom line

Every fix is a boundary decision more than a technology decision - where does a call go, who owns which data, what's sync vs. async.
The Pricing→Odoo dependency is the one We'll genuinely lose sleep over before the next seasonal spike. Everything else is worth doing but not on fire.
